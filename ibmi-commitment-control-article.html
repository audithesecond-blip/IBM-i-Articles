IBM i Commitment Control: Preventing Silent Data Corruption in Multi-File Transactions

Last Updated: April 2026

Most data corruption on IBM i is not caused by failure — it is caused by incomplete success. When your order processing program updates the order header and inventory but crashes before posting to the general ledger, you do not have a system failure. You have something worse: partial truth in your data.

This article explains why journaling alone does not prevent data corruption, how to implement commitment control correctly, and the real-world cost of getting it wrong.

🔍 The Problem: When Success Becomes Failure

Consider an order processing system that updates three files: ORDERHDR for order status, INVMAST for inventory quantities, and GLACCT for financial records. The program runs. The first two updates succeed. Then the disk fills up and GLACCT fails.

❌ Without Commitment Control
Order: COMPLETED ✓
Inventory: REDUCED ✓
Financials: MISSING ✗

Partial data persists permanently.
GL is out of balance.
✅ With Commitment Control
Order: PENDING (rolled back)
Inventory: UNCHANGED (rolled back)
Financials: NOT ATTEMPTED

No change. Data integrity preserved.

Commitment control treats multiple file operations as a single unit of work. Either all changes persist, or none do. There is no middle ground.

The Journaling Misconception

Many developers believe journaling provides transactional integrity. This is a dangerous assumption.

❌ Journaling records what happened — it does NOT undo previous updates
UPDATE ORDERHDR;  // Succeeds → Written to journal → PERMANENT
UPDATE INVMAST;   // Succeeds → Written to journal → PERMANENT
WRITE GLACCT;     // Fails   → Not written

// Result: First two updates are PERMANENT
// The journal is a history book — it cannot rewrite history
📌 The key distinction: Journaling is a history book — it records what happened. Commitment control is the decision maker — it determines what should happen, and rolls everything back if any part fails.

📚 Commitment Control Fundamentals

ACID Properties in Plain Terms

PropertyWhat It MeansWhat Breaks Without It
AtomicityAll or nothingPartial updates persist permanently
ConsistencyValid state to valid stateInvalid intermediate states in your data
IsolationChanges invisible until commitOther jobs see incomplete work-in-progress
DurabilityCommitted changes survive crashesJournaling provides this independently

The practical translation: Atomicity means you cannot have half an order. Consistency means your GL always balances. Isolation means other users never see your work in progress.

Lock Levels — Choose Wisely

LevelWhen to UseWhat It Does
*CHGStandard OLTP — 99% of casesLocks only records you change
*CSHigh-concurrency readsLocks during read, releases immediately
*ALLSerializable reportsLocks everything you touch
*NONERead-only queriesNo locking at all
Rule of thumb: Start with *CHG. Only change it if you have a specific, diagnosed problem that requires a different level.

Commitment Scope — A Critical Decision

Use *JOB scope for simple batch jobs where all programs share one transaction boundary. Use *ACTGRP scope for modular ILE applications where each activation group manages its own independent transaction.

📌 Important: You do not explicitly "start" a transaction. It starts automatically with your first database operation after STRCMTCTL. What you control is when it ends — via COMMIT or ROLLBACK.

⚙️ Setup and Configuration

Step 1 — Journal Configuration

CL — Create journal and start journaling all transactional files
/* Create journal receiver */
CRTJRNRCV JRNRCV(MYLIB/RCV0001) THRESHOLD(100000)

/* Create journal */
CRTJRN JRN(MYLIB/APPJRN)
       JRNRCV(MYLIB/RCV0001)
       MNGRCV(*SYSTEM)
       DLTRCV(*YES)

/* Start journaling all transactional files to the SAME journal */
STRJRNPF FILE(MYLIB/ORDERHDR) JRN(MYLIB/APPJRN)
STRJRNPF FILE(MYLIB/INVMAST)  JRN(MYLIB/APPJRN)
STRJRNPF FILE(MYLIB/GLACCT)   JRN(MYLIB/APPJRN)

Key parameters: THRESHOLD(100000) switches to a new receiver at 100MB. MNGRCV(*SYSTEM) lets IBM i manage receivers automatically. DLTRCV(*YES) removes old receivers after backup.

Step 2 — Program Configuration

RPGLE — All three settings are required
**FREE
CTL-OPT DFTACTGRP(*NO) ACTGRP('ORDPROC');  // Required for *ACTGRP scope

DCL-F ORDERHDR USAGE(*UPDATE) KEYED COMMIT;  // COMMIT keyword enables CC
DCL-F INVMAST  USAGE(*UPDATE) KEYED COMMIT;
DCL-F GLACCT   USAGE(*OUTPUT)       COMMIT;

Step 3 — Starting Commitment Control

CL — CL controls the boundary, RPG does the work
PGM
  STRCMTCTL LCKLVL(*CHG) CMTSCOPE(*ACTGRP)
  CALL PGM(ORDPROC)
  ENDCMTCTL
ENDPGM

This separation of concerns — CL controls the commitment boundary, RPG executes the business logic — makes testing, error handling, and future maintenance significantly easier.

🏗️ Three Implementation Patterns That Work

Pattern 1 — Basic Multi-File Transaction

RPGLE — All three files treated as one atomic unit
**FREE

DCL-F ORDERHDR USAGE(*UPDATE) KEYED COMMIT;
DCL-F INVMAST  USAGE(*UPDATE) KEYED COMMIT;
DCL-F GLACCT   USAGE(*OUTPUT)       COMMIT;

DCL-PROC ProcessOrder;
  DCL-PI *N IND;
    pOrderID CHAR(10) CONST;
  END-PI;

  COMMIT;  // Clear any previous uncommitted changes

  CHAIN pOrderID ORDERHDR;
  IF %FOUND(ORDERHDR);

    ORDSTS = 'PROCESSING';
    UPDATE ORDRHDR;

    CHAIN ORDITEM INVMAST;
    IF %FOUND(INVMAST) AND QTYOH >= ORDQTY;

      QTYOH = QTYOH - ORDQTY;
      UPDATE INVMAST;

      GLACCT_ACCOUNT = '4000-100';
      GLACCT_AMOUNT  = ORDAMT;
      WRITE GLACREC;

      ORDSTS = 'COMPLETED';
      UPDATE ORDRHDR;
      COMMIT;      // All three operations committed together

      RETURN *ON;
    ELSE;
      ROLLBACK;  // None of the above persists
    ENDIF;
  ENDIF;

  RETURN *OFF;
END-PROC;

If the WRITE to GLACCT fails, ROLLBACK undoes both previous UPDATEs. Your data never enters an inconsistent state.

Pattern 2 — Production-Ready Error Handling

RPGLE — MONITOR/ON-ERROR guarantees rollback on any failure
**FREE

DCL-F ORDERHDR USAGE(*UPDATE) KEYED COMMIT;
DCL-F INVMAST  USAGE(*UPDATE) KEYED COMMIT;
DCL-F GLACCT   USAGE(*OUTPUT)       COMMIT;
DCL-F ERRLOG   USAGE(*OUTPUT);  // NOT under commitment control — always written

DCL-PROC ProcessOrderSafe;
  DCL-PI *N IND;
    pOrderID CHAR(10) CONST;
  END-PI;

  DCL-S Success IND INZ(*ON);

  MONITOR;
    COMMIT;

    CHAIN pOrderID ORDERHDR;
    IF NOT %FOUND(ORDERHDR);
      Success = *OFF;
      LEAVE;
    ENDIF;

    ORDSTS = 'PROCESSING';
    UPDATE ORDRHDR;

    CHAIN ORDITEM INVMAST;
    IF NOT %FOUND(INVMAST) OR QTYOH < ORDQTY;
      Success = *OFF;
      LEAVE;
    ENDIF;

    QTYOH = QTYOH - ORDQTY;
    UPDATE INVMAST;

    GLACCT_AMOUNT = ORDAMT;
    WRITE GLACREC;

    COMMIT;

  ON-ERROR;
    ROLLBACK;           // Catches ANY error — guaranteed rollback
    Success = *OFF;

    // Error log is outside commitment control
    // Always written regardless of transaction outcome
    ERRLOG_TIME  = %TIMESTAMP();
    ERRLOG_ORDER = pOrderID;
    WRITE ERRLOGREC;
  ENDMON;

  RETURN Success;
END-PROC;
📌 Key design decision: The error log file is deliberately NOT under commitment control. This ensures you always have a record of what went wrong, even when the transaction rolls back.

Pattern 3 — High-Volume Batch Processing

RPGLE — Commit every 1,000 records to balance performance and recovery
**FREE

DCL-C COMMIT_INTERVAL 1000;
DCL-S Processed PACKED(7:0) INZ(0);
DCL-S Committed PACKED(7:0) INZ(0);

DCL-PROC BatchProcess;

  COMMIT;

  SETLL *START ORDERHDR;
  READ ORDERHDR;

  DOW NOT %EOF(ORDERHDR);

    IF ORDSTS = 'PENDING';
      MONITOR;
        ProcessSingleOrder(ORDID);
        Processed += 1;

        IF %REM(Processed: COMMIT_INTERVAL) = 0;
          COMMIT;                    // Commit every 1,000 records
          Committed = Processed;
        ENDIF;

      ON-ERROR;
        ROLLBACK;                    // Only current batch rolls back
        Processed = Committed;       // Reset counter to last safe point
        LogError(ORDID);
      ENDMON;
    ENDIF;

    READ ORDERHDR;
  ENDDO;

  IF Processed > Committed;
    COMMIT;  // Final commit for remaining records
  ENDIF;

END-PROC;

Committing every 1,000 records balances performance (fewer commits means faster processing) against recovery time (smaller batches means less to redo on failure). If an error occurs, only the current batch of up to 1,000 records rolls back — not the entire job.

🔗 Mixing Native I/O and SQL

When a program uses both native RPG I/O and embedded SQL, both must share the same transaction boundary. The SET OPTION COMMIT directive tells SQL to participate in the same commitment control context as native I/O.

RPGLE — Native I/O and SQL in the same transaction
**FREE

DCL-F ORDERHDR USAGE(*UPDATE) KEYED COMMIT;

EXEC SQL SET OPTION COMMIT = *CHG;  // SQL joins the same transaction

DCL-PROC ProcessOrderMixed;

  COMMIT;

  // Native I/O
  CHAIN pOrderID ORDERHDR;
  ORDSTS = 'PROCESSING';
  UPDATE ORDRHDR;

  // SQL in the same transaction
  EXEC SQL
    UPDATE INVMAST
    SET QTYOH = QTYOH - :ORDQTY
    WHERE ITEMID = :ORDITEM;

  IF SQLSTATE <> '00000';
    ROLLBACK;  // Rolls back BOTH the native I/O and the SQL
    RETURN *OFF;
  ENDIF;

  EXEC SQL
    INSERT INTO GLACCT (ACCOUNT, AMOUNT)
    VALUES ('4000-100', :ORDAMT);

  IF SQLSTATE <> '00000';
    ROLLBACK;
    RETURN *OFF;
  ENDIF;

  COMMIT;  // Commits BOTH native I/O and SQL together
  RETURN *ON;

END-PROC;

Journal Receiver Sizing

The sweet spot for most environments is 100MB to 1GB depending on transaction volume. Too small (10MB) causes the system to switch receivers every hour, degrading performance. Too large (5GB) means longer recovery time after a crash.

Lock Escalation — The Hidden Performance Killer

❌ One massive transaction — locks 100,000 records for hours
COMMIT;
SETLL *START ORDERHDR;
READ ORDERHDR;
DOW NOT %EOF(ORDERHDR);
  UPDATE ORDERHDR;  // Lock held until final COMMIT
  READ ORDERHDR;
ENDDO;
COMMIT;  // 100,000 locks released here — after hours
✅ Frequent commits — locks released every 1,000 records
COMMIT;
DCL-S Counter INT(5) INZ(0);

SETLL *START ORDERHDR;
READ ORDERHDR;
DOW NOT %EOF(ORDERHDR);
  UPDATE ORDERHDR;
  Counter += 1;

  IF Counter >= 1000;
    COMMIT;      // Locks released — other jobs can proceed
    Counter = 0;
  ENDIF;

  READ ORDERHDR;
ENDDO;

IF Counter > 0;
  COMMIT;  // Final commit for remaining records
ENDIF;

🚨 Four Common Pitfalls

Pitfall 1
The Forgotten COMMIT

A program updates a file under commitment control, then ends without issuing a COMMIT. IBM i automatically rolls back all uncommitted changes when the program ends. Your update is silently lost — no error, no warning.

The fix: Always issue an explicit COMMIT before *INLR = *ON. Make it the last line of every procedure that writes data.

Pitfall 2
Mixed Commitment Scopes

Program A uses ACTGRP('GROUPA') and Program B uses ACTGRP('GROUPB'). They run in the same job but maintain independent transactions. A COMMIT in Program A does not commit Program B's changes — and vice versa.

The fix: Programs that share a transaction must use the same activation group name.

Pitfall 3
Long-Running Transactions
❌ Lock held for entire user interaction — could be minutes
COMMIT;
CHAIN OrderID ORDERHDR;
UPDATE ORDRHDR;   // Lock acquired here
EXFMT ORDERSCR;  // User stares at screen for 10 minutes
UPDATE ORDRHDR;
COMMIT;           // Lock finally released — 10 minutes later
✅ User interaction outside the transaction — lock held for milliseconds
CHAIN OrderID ORDERHDR;
EXFMT ORDERSCR;   // User interaction OUTSIDE the transaction
COMMIT;
CHAIN OrderID ORDERHDR;
UPDATE ORDRHDR;   // Lock acquired and released quickly
COMMIT;
Pitfall 4
Missing ROLLBACK in Error Handler
❌ No ROLLBACK — FILE1 gets committed even though FILE2 failed
MONITOR;
  COMMIT;
  UPDATE FILE1;
  UPDATE FILE2;  // Fails
  COMMIT;
ON-ERROR;
  // ROLLBACK missing — FILE1 remains uncommitted
ENDMON;
COMMIT;  // Commits FILE1 even though FILE2 failed
✅ ROLLBACK in error handler — all or nothing guaranteed
MONITOR;
  COMMIT;
  UPDATE FILE1;
  UPDATE FILE2;
  COMMIT;
ON-ERROR;
  ROLLBACK;  // Required — undoes FILE1 if FILE2 failed
ENDMON;

📊 Monitoring and Troubleshooting

Check for Uncommitted Changes

CL — Identify uncommitted journal entries
DSPJRN JRN(MYLIB/APPJRN)
       ENTTYP(*ALL)
       CMTCYCID(*UNCOMMITTED)
       OUTPUT(*PRINT)

/* If entries appear here, a program forgot to COMMIT */

Check Lock Status

CL — Identify jobs holding locks
WRKOBJLCK OBJ(MYLIB/ORDERHDR) OBJTYPE(*FILE)

/* Locks held for minutes or hours indicate long-running transaction problems */

Key Messages to Monitor

MessageMeaningAction Required
CPF8355COMMIT completedNormal — no action
CPF8356ROLLBACK completedInvestigate the cause
CPF8358Commitment control not activeCheck STRCMTCTL configuration

Frequent CPF8356 messages indicate application errors or business rule violations occurring in production. Each one deserves investigation.

📌 After a crash: IBM i's recovery is automatic. The system detects abnormal termination, rolls back all uncommitted changes, releases all locks, and marks journal entries accordingly. You do not need to manually fix data after a crash — commitment control handles it.

📈 Two Real-World Case Studies

Case Study 1 · Regional Bank · GL System

The problem: Month-end GL posting showed $50K–$200K discrepancies. Manual reconciliation took 2–3 days every month.

Root cause: Three GL files updated without commitment control. When any update failed mid-sequence, previous updates persisted permanently — leaving the GL out of balance.

The fix: All three GL files placed under the same commitment boundary with MONITOR/ON-ERROR error handling and automatic rollback on failure.

Results after 18 months:
  • Zero data integrity issues
  • Month-end close time reduced from 3 days to 4 hours
  • Passed audit with no findings
  • Estimated savings: $150,000 per year
Case Study 2 · E-Commerce · Inventory Overselling

The problem: During flash sales, inventory went negative. The system sold 500 units when only 300 were in stock.

Root cause: Multiple jobs simultaneously passed the inventory check, each seeing the same available quantity before any of them had committed a reduction.

The fix: COMMIT issued before the CHAIN operation, which acquires a record lock before the availability check. Only one job can hold that lock at a time — serialising access and preventing the race condition.

Results:
  • Zero overselling incidents since implementation
  • Cancelled orders reduced by 95%
  • Customer satisfaction scores improved measurably

✅ Best Practices Checklist

Configuration
  • Journal receivers sized appropriately — 100MB to 1GB
  • MNGRCV(*SYSTEM) enabled for automatic receiver management
  • All transactional files journaled to the same journal
Programming
  • DFTACTGRP(*NO) in all ILE programs using commitment control
  • COMMIT keyword on all file declarations under commitment control
  • Explicit COMMIT before *INLR = *ON in every procedure that writes data
  • ROLLBACK present in every ON-ERROR handler
  • Consistent activation group names across programs sharing a transaction
  • Periodic commits in batch jobs — every 1,000 to 10,000 records
Architecture
  • Transaction boundaries documented and reviewed
  • Short transactions — no user interaction inside a commit boundary
  • Error logging files outside commitment control
Operations
  • Journal receiver switching monitored
  • Lock contention reviewed regularly using WRKOBJLCK
  • CPF835x messages reviewed and investigated
  • Rollback scenarios tested before production deployment

💡 Conclusion

Commitment control is not optional — it is the only protection against partial truth in your data. The 4–20% performance overhead is negligible compared to the cost of manual reconciliation, data corruption, audit findings, and production incidents.


Three principles to apply consistently:

  • All or nothing — related updates must commit or rollback together
  • Explicit control — always explicitly COMMIT or ROLLBACK, never rely on implicit behaviour
  • Short transactions — do not hold locks longer than necessary

If you do not control your transaction boundaries, your system will define them for you — and not in your favour.
